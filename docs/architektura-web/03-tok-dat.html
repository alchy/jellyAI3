3 · Tok dat systémem

Dvě cesty dat: offline build korpusu (jednou) a runtime tah dialogu (pořád dokola). Tato kapitola je mapa — detaily jednotlivých zastávek mají vlastní kapitoly.

3.1 Offline: build korpusu

Korpus (data/raw/*.txt) se jednorázově přemění na faktový graf (data/graph.pkl). Tady — a jen tady — pracují statistické nástroje ÚFAL.

flowchart LR
    RAW["data/raw/*.txt"] --> IDX["index<br/>(dokumenty, pasáže)"]
    IDX --> ANN["annotate<br/>NameTag (entity)<br/>UDPipe (tokeny, lemmata, deprely)<br/>po VĚTÁCH"]
    ANN --> EXT["extrakce faktů<br/>spony · apozice · koordinace<br/>pro-drop · aliasy „řečený“"]
    EXT --> HYG["hygiena<br/>korpusová hlasování<br/>(upos, pád, životnost, lemma)"]
    HYG --> CAN["kanonizace<br/>+ poziční merge<br/>+ nominativizace id"]
    CAN --> INS["instanční vrstva<br/>(srůst jmen JEN z tvrzení)"]
    INS --> PKL["data/graph.pkl"]

Orchestraci nese build_fact_graph (jellyai/tasks.py:40) — skutečný výřez:

# jellyai/tasks.py:50–75 (zkráceno)
dropped_e = scrub_entities(annotations, form_case_votes(annotations))
graph = build_graph(annotations)
recover_entities(annotations, graph)   # role ②: doplnit tituly, co NER minul
resolve_entities(graph)
dropped_p, dropped_f = scrub(graph, lemma_upos_votes(annotations))
dropped_s = scrub_semantics(graph, noun_animacy_votes(annotations))
renamed = nominativize(graph, propn_lemma_votes(annotations))
merged = resolve_instances(graph)
graph.save(config.graph.graph_path)

Proč tolik „hygieny“: tagger v jedné větě lže (mis-tagy VERB/NOUN, špatné pády). Rozhoduje proto vždy hlasování tvaru přes celý korpus — to je zákon 5 projektu (kapitola 16). Detail buildu v kapitole 6, lingvistika v kapitole 4.

3.2 Runtime: jeden tah dialogu

Každý vstup uživatele (otázka, výrok, volba, příkaz) prochází IrisAutomaton.turn(). Zjednodušený, ale věrný sled:

sequenceDiagram
    autonumber
    participant U as uživatel
    participant I as IrisAutomaton
    participant C as claims (přímí experti)
    participant M as Mnemos (zápis)
    participant A as GraphAnswerer
    participant F as FactGraph
    participant K as karty (deck)
    U->>I: text tahu
    I->>I: fire_due() — dozrálé připomínky
    I->>C: recognize(text)? (Metron, Chronos-hodiny, meta-focus)
    alt expert si tah nárokuje
        C-->>U: přímá odpověď (výpočet / hodiny / těžiště)
    else běžný tah
        I->>I: volby rozpracovaného dialogu (identita, kandidát)
        alt text bez „?“ (výrok/příkaz)
            I->>M: parse_clauses → karty statement.* → fakt(y)
            M->>F: add_fact + deník memory.jsonl
            M-->>U: „Zapamatováno …“
        else otázka
            I->>A: answer(text)
            A->>A: build_query — vzorové karty → pseudo-QL Pattern
            A->>F: match: neúplný fakt → díra
            F-->>A: hodnoty + fakt + stopa
            A-->>I: odpověď / přetečení / nenašel
            I->>K: událost tahu (resolve.ambiguous / data.overflow / focus.low)
            K-->>U: odpověď NEBO doostřovací dialog
        end
    end

Skutečné pořadí větví v _turn() (jellyai/iris/automaton.py:199): přímí experti smyčkou nad claimy (řádek 204) → volba identity podmětu (#43) → volba kandidáta z nabídky → větev bez otazníku (připomínky, zapomínání, „zapamatuj si“, správa plánu, posílání, focus-shift, konstatování) → vzpomínání a dotaz na plán → zahřátí časového intervalu → vlastní odpověď answereru → dialogové karty. Kapitola 11 ukazuje, jak z tohoto sledu vzniká grafový cyklus.

3.3 Cesta otázky (detail)

Otázka „Kdo napsal R.U.R.?“ projde těmito stanicemi:

flowchart TB
    T["„Kdo napsal R.U.R.?“"] --> LEX["lexer.classify<br/>kdo{otaz} · napsal{l_tvar} · R.U.R.{jmeno}"]
    LEX --> CARD["výběr vzorové karty<br/>q-otaz-minuly: [%{TAZACI}, ?%{SE}, %{SLOVESO_MINULE}, %{ENTITA}]"]
    CARD --> PAT["pseudo-QL Pattern<br/>predikát: napsat · známý: R.U.R. (obj)<br/>díra: subj / person"]
    PAT --> RES["rozřešení entit<br/>„R.U.R.“ → uzel R.U.R. (orákulum grafu)"]
    RES --> MATCH["_match nad fakty<br/>napsat(subj: ?, obj: R.U.R.)"]
    MATCH --> ANS["odpověď: Karel Antonín Čapek<br/>+ stopa: téma → fakt → hodnota"]
    ANS --> GLOW["aktivace: odpověď a entity svítí<br/>(těžiště pro další tah)"]

Aktivační okno po tomto tahu (skutečné hodnoty z provozu):

Karel Antonín Čapek = 2.68   ← odpověď svítí nejvíc
R.U.R.              = 2.10   ← rozřešená entita otázky
Josef Čapek         = 0.51   ← vyzářeno po hranách (bratr)
člověk, bratr       ≈ 0.35   ← okolí tématu

Následující tah z jasu těží: „Kdy se narodil?“ nemá podmět, ale nejteplejší rodově shodná osoba je Karel Čapek — systém doplní podmět z těžiště (pro-drop na straně dotazu) a odpoví. Aktivace není ozdoba, ale nosný mechanismus.

3.4 Cesta výroku (zápis paměti)

„Marcela bydlí v Petrovicích.“ nemá otazník — jde zápisovou větví:

flowchart TB
    S["„Marcela bydlí v Petrovicích.“"] --> CL["parse_clauses<br/>(souvětí = fakt na klauzuli)"]
    CL --> LEX2["lexer + vzorové karty výroků<br/>(utterance.statement)"]
    LEX2 --> KIND["druh výroku kartou:<br/>prézentní děj s místem"]
    KIND --> NOM["nominativizace<br/>Petrovicích → Petrovice (UDPipe + toponymní pojistka)"]
    NOM --> FACT["fakt: bydlet(subj: Marcela, loc: Petrovice,<br/>theme: uživatel, time: 17. 7. 2026 12:00)"]
    FACT --> STORE["graf (add_fact) + deník data/memory.jsonl<br/>(append-only, po restartu se přehraje)"]
    STORE --> OK["💬 Zapamatováno."]

Klíčové: fakt jde do téhož grafu jako korpusová znalost, jen nese timestamp dialogu a pozorovatele (theme = uživatel). Otázka „Kde bydlí Marcela?“ pak jede toutéž dotazovou cestou jako otázky nad korpusem — a nový predikát z výroku rozšíří slovník dotazového parseru: co se zapíše, na to se lze hned ptát.

3.5 Tři grafy jednoho tahu

Tok dat lze shrnout i takto — v každém tahu se potkají tři grafy s různou životností (koncept z spec otázkového grafu):

GrafŽivotnostObsahRole v toku
větnýjeden tahtokeny + hypotézy tříd + nároky expertůvstupní reprezentace; osvětluje otázkový graf
otázkovýtrvalý (kompilát)typy otázek + workeři + zpřesňovací hranysměrování tahu, vedení dialogu
faktovýtrvalý (roste)fakty + aktivační polezdroj odpovědí, konverzační těžiště

Dialog jako cyklus: větný graf osvětlí otázkový → vítězný uzel + dekorace → dotaz do faktového → výsledek → text → neosvětlené role a alternativy = hrany zpět do otázkového grafu → další tah. Podrobně v kapitole 11.
