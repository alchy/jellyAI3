15 · Workflow systému

Provozní a vývojové smyčky: build korpusu, služby, benchmarky jako brány commitů, telemetrie a triage.

15.1 Build korpusu

data/raw/*.txt  →  ./jelly index  →  ./jelly annotate  →  ./jelly graph
                   (dokumenty)       (ÚFAL služby         (extrakce + hygiena +
                                      :8081–8083)           kanonizace + instance
                                                            → data/graph.pkl)

Vlastní korpus: vhoď .txt do data/raw/ a spusť řetěz. Po přestavbě grafu se smí id uzlů legitimně zlepšit (Betlémě→Betlém) — očekávání benchmarků se pak aktualizují se zdůvodněním v commitu; nikdy se neaktualizují, aby zakryla regresi. Namátková kontrola známých uzlů (Ježíš, Božena Němcová, Betlém) po každé přestavbě je součást pracovní smyčky.

15.3 Benchmarky: pět bran commitu

Před každým commitem musí projít celá sada (HANDOVER §3):

.venv/bin/python -m pytest -q                # N passed, 0 failed
.venv/bin/python benchmark/run_etalon.py     # jádro 100 %
.venv/bin/python benchmark/run_focus.py      # 12/12
.venv/bin/python benchmark/run_dialog.py     # 100 %
.venv/bin/python benchmark/run_mnemos.py     # zápis 100 %
.venv/bin/python benchmark/run_coverage.py   # diagnostika (trend)
BenchmarkCo měříJak
run_etalon.pyodpovědi na korpusem kryté otázkyřádky {"q", "expect", "cat"}; negativa přes "reject"
run_focus.pyaktivace správných uzlůočekávané uzly v top-K jasu
run_dialog.pyscénáře automatufixní hodiny datetime(2026,7,17,12,0), čerstvý graf per scénář — determinismus
run_mnemos.pyzápis: výrok → parsetýmž soukolím jako runtime; --nom měří i nominativizaci (chce běžící ÚFAL)
run_coverage.pypodíl vět korpusu bez faktutrend, ne normativ
run_qgraph.pyshadow parita otázkového grafu4 roviny: dispatch dialog/etalon, stav, dekorace; počítá remízy

Gap řádky: institucionalizovaná poctivost

Známý nedostatek se nemaže — přidá se mu "gap": "proč + odkaz na backlog". Benchmark ho reportuje zvlášť a hlídá, že se nezhorší; jakmile projde, ohlásí GAP-FIXED — a jakmile ho něco znovu rozbije, číslo klesne. Nová feature bez svého benchmarkového řádku se nepovažuje za hotovou. Vzorový životní cyklus (z experimentu E4): scénář založen červený → mechanismus → GAP-FIXED → řádek zůstává jako regresní hlídka.

    F --> G["7 · ověř E2E přes REST/web<br/>to, co uvidí uživatel"]

Past 4: nikdy nespouštěj pytest přes | tail v řetězu s && — maska exit kódu už jednou pustila SyntaxError do main. Nejdřív celý běh, pak čti. Past 7: commit zprávy s českými uvozovkami přes heredoc (git commit -F - <<'MSG').

15.5 Telemetrie a triage: učení z provozu

V provozu (ne v testech) píše automat stopu každého tahu do data/telemetry.jsonl: ts, otázka, druh, odpověď, assurance, komponenty, karty (včetně vzorové karty dotazu), téma a predikát trasy. Nad stopou pracuje ./jelly triage: miss = ne-answer / assurance < 0.5 / jazykový marker (miss_markers v cs.json — i texty terminálů jsou data); shlukuje se podle (druh, karta), největší shluk první, s ukázkami otázek. Výstup triage je přirozený zdroj nových etalon řádků a kandidátů na nové karty — provoz plní benchmarky průběžně.

15.6 Shadow režim: jak se zavádějí nové koncepty

Workflow projektu pro větší architektonické změny (vzor: otázkový graf):

  1. Spec s mentálním modelem — koncept se nejdřív „projde“ na skutečných tazích (spec 2026-07-20-otazkovy-graf.md).
  2. Kompilace k nahlédnutí — nová struktura se postaví vedle staré, bez změny chování (./jelly qgraph dump).
  3. Shadow harness — měří shodu nové cesty se starou na existujících benchmarcích (parita), plus nové gap řádky na chování, které stará cesta neuměla.
  4. Kritérium přijetí: parita drží (nic pod 100 %) a aspoň jeden nový řádek zezelená. Co kritérium nesplní, se nepřijímá — zůstane zapsaným nálezem (vzor: váhy z telemetrie, změřená mrtvá větev).
  5. Přepnutí — samostatné rozhodnutí s bitovou paritou (vzor: dispatch přímých expertů, fáze D).
