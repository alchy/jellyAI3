7 · Otázkový graf

Druhý trvalý graf systému: ne parser, ale graf možných interpretací otázky — typy otázek jako uzly, digging jako hrany, workeři jako atributy a osvětlení jako mechanismus výběru.

7.1 Co otázkový graf je (a co není)

Intuice: když přijde otázka, systém se neptá „jak ji rozparsovat?“, ale „která ze známých interpretací otázky právě svítí?“. Otázkový graf je mapa všech způsobů, kterými se lze systému ptát: každý uzel je jeden typ otázky („kdo-udělal-co“, „zjišťovací existence“, „aritmetický výpočet“, „hodinová otázka“), a hrany říkají, kudy smí dialog legitimně pokračovat (zpřesnění při přetečení, návrat po volbě). Věta tahu uzly osvětlí; nejsilněji osvětlený uzel tah směruje.

Co otázkový graf není:

  • Není to parser. Parsování (match vzoru na tokeny) zůstává mechanismem matcheru; graf jen strukturuje, co se matchuje a kam vede výsledek.
  • Není to druhá pravda. Graf je deterministický kompilát karet + schématu predikátů: zdroje se editují, graf se přegenerovává. Karty jsou zdrojový kód, graf jejich zkompilovaná podoba — proto se nesmí rozejít a proto je jeho bránou parity gate (shadow harness, §7.7).
  • Není to ATN. Hrany nenesou podmínky ani program — jen typ přechodu. Rekurzivní přechodové sítě s podmínkami jsou vědomě odmítnutá historická past (kapitola 16).

7.3 Hrany: odvozené, ne kreslené

Hrany zpresneni (otázka → clarify uzel) a navrat (clarify → přehrání otázky) se odvozují z dat karet, nikoli enumerují ručně — a odvození není kartézský součin. Dvě pravidla (výsledek experimentu E4 — počet hran klesl z 80 na 58):

# jellyai/iris/qgraph.py:88–96
for node in nodes.values():
    if node.kind == "otazka":
        # digging hrany se ODVOZUJÍ z dat karet (#57 E4), žádný
        # kartézský součin: statement clarify z otázky nevede,
        # přetečení dává smysl jen u výčtových děr
        node.edges = [
            QEdge("zpresneni", name) for name, event in clarify_events
            if not event.startswith("statement.")
            and (event != "data.overflow" or has_hole.get(node.card))]

Tedy: zpřesnění identity podmětu výroku (statement.subject) z otázky nevede (je to výroková záležitost) a hrana na overflow nabídku vede jen z otázek s dírou (zjišťovací otázka nemá co přetékat). Drill hrany (navazující „Kdy?“ po odpovědi) se na úrovni grafu neenumerují vůbec — počítají se za běhu z rolí žhavého faktu (kapitola 13).

7.4 Osvětlení: dva druhy rozsvěcení

Klíčové rozlišení celého modelu, potvrzené měřením: rozsvěcení má dva druhy.

Soutěžící uzly — jeden vyhrává tah

illuminate (qgraph.py:191) rozsvítí uzly, které na větu tahu sedí, a seřadí je klíčem (tier, priorita, délka vzoru): tier 3 = přímý expert (nárok na obsah řádku — Metronův výraz, hodinová otázka, meta-fráze), tier 2 = vzorové uzly otázek. Je to týž klíč, jakým vybírá deck — graf nezavádí druhé rozhodování, jen dává plochému výběru strukturu.

7.5 Ukázky průchodu

A — jednoduchá otázka (jeden uzel svítí)

„Kdo napsal R.U.R.?“

větný graf:    kdo{otaz}   napsal{l_tvar}   R.U.R.{uzel+ ✓ orákulum}
                  │
otázkový graf: [q-otaz-minuly] svítí (jediný)      worker = graf
                  │                                 dekorace: —
faktový graf:  _match(napsat, {R.U.R.}, díra subj/person)
                  │
odpověď:       Karel Antonín Čapek

B — soutěž uzlů + dekorace

„Co řekl Ježíšovi?“

větný graf:    co{otaz}   řekl{l_tvar}   Ježíšovi{dativ ⇒ hypotéza adresáta}
                  │
otázkový graf: svítí DVA uzly, vyhrává těsnější:
                  [q-rekl-adresatovi]  (priorita 6)   ← vítěz
                  [q-otaz-minuly]      (priorita 4)
                  │
dekorace:      role:adresat    ← NEsoutěží, věší se na vítěze
                  │
faktový graf:  _match(říci, {Ježíš}, díra obj) + rolová vazba theme
                  │           └ známý MUSÍ být v roli theme (adresát)
odpověď:       „umřel“   (Martina slova Ježíšovi — ne Ježíšova vlastní)

C — dialog jako procházka grafem

Nejsložitější případ nemá jednu odpověď — má jich 143:

❓ Co řekl Ježíš? — uzel otázky svítí, faktový graf vrací PŘETEČENÍ → hrana zpresneni → uzel focus-offer-overflow

💬 Mám tu více dat… Nabízí se: Jeremiáš, Petr, bůh, celník, chleb…

❓ Petr — stojíme ve zpřesňovacím uzlu; volba = hrana navrat; otázka se PŘEHRAJE s dominantní oblastí (pick_focus)

💬 satane, nemohli jedinou hodinu bdít se mnou

Zpřesňovací uzly jsou plnohodnotné uzly grafu: je-li takový uzel aktivní, dialog pokračuje zpřesňováním a graf si sám ostří focus.

7.6 Dialogové stavy: pozice v grafu

Stav dialogu byl historicky roztroušený po objektech (PendingFocus, drill přes _prev_trace, pick_focus). Model otázkového grafu je sjednocuje do jediné věci: pozice v grafu — DialogPosition (qgraph.py:151):

class DialogPosition:
    def enter(self, name):    # vstup do uzlu (tah směrovaný osvětlením)
    def sharpen(self, name):  # krok po zpřesňovací hraně; False = hrana neexistuje
    def resume(self):         # návrat ze zpřesnění: cíl hrany `navrat`
                              # („*“ = přehraj rozpracovanou otázku)

Shadow harness ověřuje, že sekvence enter → sharpen → resume reprodukuje skutečné chování PendingFocus na dialogových scénářích (stav 3/3). Plné nahrazení pending objektů pozicí je další fáze experimentu.

7.7 Kompilace, parity gate a co experiment naměřil

Shadow harness benchmark/run_qgraph.py chování neposouvá — porovnává, kudy by šel graf, s tím, kudy systém reálně šel:

RovinaVýsledekCo to znamená
Dispatch (dialog)11/11 (100 %)osvětlení směruje jako ruční pořadí bran
Dispatch (etalon)50/50 (100 %)vítězný uzel = karta, kterou reálně vybral build_query
Výroky15/15 (100 %)výroková rodina (#51): vítěz osvětlení = skutečná větev zápisu
Stav dialogu4/4 (100 %)zpřesnění + návrat hranou reprodukuje PendingFocus
Dekorace50/50 (100 %)předpověď nároků = skutečně aplikované filtry
Remízy výběrového klíče0klíčový nález — viz níže

Čtyři nálezy, které stojí za zapamatování:

  1. Dekorace jsou popis reality. To, co model předpovídá, answerer už dělá — zelená pro formální claim() rozhraní (#26).
  2. Váhy z telemetrie jsou mrtvá větev. Otázka „kolik provozu je potřeba, než váhy něco znamenají?“ má dřívější otázku: nastávají vůbec remízy? Naměřeno 0 remíz základního klíče (priorita × délka vzoru) na celém dialogu i etalonu — váhy nemají kde zasáhnout, při žádném objemu provozu. Zůstávají jako diagnostika; harness remízy počítá dál.
  3. Hranice rozsahu je ostrá: 36 z 50 dialogových tahů je mimo dotazovou polovinu (výroky, příkazy, připomínky). Výroková typologie je jiný graf — nekřížit předčasně (vědomá hranice #51, kapitola 8.5).
  4. 100 % shody je silný i slabý signál. Silný: kompilace je věrná. Slabý: graf zatím nedělá jiné rozhodnutí — zisk není v přesnosti, ale ve struktuře (jedno místo pro směrování, hrany pro digging, jedna reprezentace stavu). Hodnotu prokázaly až fáze, kde dřívější kód neměl odpověď: chytrá clarifikace ze schématu (kapitola 10) a proaktivní nabídky (kapitola 13).
