# Spec: pseudo-QL — šablonový dotazovací jazyk nad faktovým grafem

> **Handoff pro instanci Fable.** Cíl: dotazování bez UDPipe. Korpus staví graf;
> otázka se **šablonami** přeloží na pseudo query language (`Pattern`), který se
> vykoná nad grafem. Základ existuje (`jellyai/answerer/query.py`, 6 testů,
> **vypnutý** `use_templates=False`), primární cesta je zatím UDPipe. Úkol:
> **dovést šablony na paritu (etalon 24/24 s `use_templates=True`) a přepnout.**

## 1. Mechanismus a princip

```
otázka (řetězec, končí "?")
   │  identifikace vzoru (šablony nad slovníkem grafu + jazykové tabulky)
   ▼
pseudo-QL  =  Pattern(predicate, known=[(role, term|SubQuery)], hole_role, hole_type, date_part)
   │  vykonání (GraphAnswerer._pattern_answer — BEZE ZMĚNY)
   ▼
odpověď z grafu (účastník v roli díry, ranking váha + aktivace)
```

**Nosné pravidlo: slovník dotazu = SÁM GRAF.** Predikáty jsou množina
`{fact.predicate}`; entity jsou `graph.nodes`. Nic se nehádá z externího
korpusu — diakritika ani mis-tagging ML parseru (UDPipe „jezis"→ADJ) nehrají
roli. Jazykové tabulky (`jellyai/lang/cs.json`) dodají jen funkční slova.

## 2. Datový model, do kterého se dotazuje (neměnit)

- **Uzel** `Node(id, type, weight)`; typy `person/geo/time/number/concept/
  institution/dílo/výrok`.
- **Fakt** `FactNode(predicate, participants, weight, source)`; `participant =
  (role, node, type)`; role `subj/obj/loc/time/num/pred/attr/theme/val`.
- Predikát je lemma slovesa, `být` (identita/vlastnost), `druh` (apoziční
  zařazení), `kontext` (asociace), `bratr`… (reifikovaný vztah), `rok/měsíc/den`
  (dekompozice data).
- `graph.facts_of(node, role=, predicate=)`, `graph.participants(fact, role)`.

## 3. pseudo-QL = `Pattern` / `SubQuery` (sdílené s `pattern.py`, NEMĚNIT tvar)

```python
Pattern(predicate:str|None, known:list[(role:str, term:str|SubQuery)],
        hole_role:str|None, hole_type:str|None, date_part:str|None)
SubQuery(predicate:str, known:list[(role, term|SubQuery)], hole_role:str="subj")
```

Sémantika (jak to vykoná existující `_pattern_answer`/`_match`/`_solve`):
- `known` termíny se rozřeší `_resolve_topic` (fuzzy: exact > case-insens >
  kmen > bezdiakritika > afinita) — **term může být skloněný/bez diakritiky**.
- `SubQuery` = rekurzivní pod-dotaz (řeší `_solve`): nejdřív vyřeš jeho `known`,
  pak `_match` → uzel; ten je `known` nadřazeného. Hloubka libovolná.
- `hole_role`/`hole_type` = preference díry (ne filtr); `_match` vrátí účastníka
  faktu, který NENÍ v `known` (symetrie vztahů funguje sama).
- `date_part` (`rok/měsíc/den`) = 2-skokový drill přes časový uzel.

## 4. Požadavky na parser `build_query(question, predicates, …) -> Pattern|None`

Vrátí `None`, když věta není otázka (bez „?") NEBO vzor nelze bezpečně
sestavit → volající spadne na UDPipe fallback. **Nikdy nevracet chybný
Pattern** (to je horší než None — fallback nenaskočí). To je hlavní příčina
současných regresí (17/24).

### 4.1 Identifikace typu dotazu (díra)
Tázací slovo (bezdiakritické) → `(hole_role, hole_type)` z tabulky `_HOLE`:
`kdo→(subj,person)`, `co→(obj,None)`, `kde/kam→(loc,geo)`, `kdy→(time,time)`,
`kolik→(num,number)`, `jaký/jakou/který→(attr,None)`, `čí→(subj,person)`.

### 4.2 Identifikace predikátu
- Slovesný tvar → predikát grafu **prefixem kmene** (napsal≡napsat,
  narodil≡narodit): sdílený počáteční kmen, koncovka se liší. Práh ≥4 znaky,
  pokrytí většiny kratšího. **Guard:** kandidát na sloveso musí být malými
  písmeny (velké = vlastní jméno; „Němec" ≠ „neměnit").
- Synonyma predikátů z `lang.predicate_synonyms` (bydlet≡žít).
- Sponová věta (`_COPULA`: je/byl/byla/…) bez vztahového jména → predikát `být`.
- Vztahové jméno (`lang.relational_nouns`: bratr/matka/…) → predikát = to jméno.

### 4.3 Identifikace předmětu (entita)
- Entita = běh obsahových tokenů, který se **rozřeší na uzel grafu**. Slovník
  entit je graf: použij přísný test `is_node(span)` = rozpětí se rozřeší A jeho
  kmenová/bezdiakritická slova jsou podmnožinou slov uzlu.
- **Greedy longest-match dělení slepených běhů** (POŽADAVEK — dnes chybí):
  „Napsal Karel Čapek **Válku s mloky**" → dvě entity „Karel Čapek" + „Válka
  s Mloky", ne jeden slepenec. Rozděl běh na maximální `is_node` rozpětí.
- Role známé entity = komplement díry: díra `subj` → entita `obj`; jinak entita
  je `subj` (téma). U identity `být` je entita `subj`.

### 4.4 Rekurze
- Vztažné „který/která/…" (`_RELATIVE`) → `SubQuery` z následujícího
  „PREDIKÁT PŘEDMĚT": „bratr autora **který napsal R.U.R.**" →
  `bratr(obj=SubQuery(napsat, obj=R.U.R.))`.
- Vztahové jméno + genitivní entita → `SubQuery` (nebo přímý `known obj`);
  „bratr Karla Čapka" → bratr s obj=Karel Čapek.

### 4.5 Zjišťovací (ano/ne) otázky — POŽADAVEK (dnes chybí)
Věta bez tázacího slova, začínající slovesem („Napsal Karel Čapek X?") →
`Pattern` s `hole_role=None`; existující answerer to vykoná jako existenční
test (`_existence` → „Ano"/nenašel). Predikát + všechny entity jako `known`.

### 4.6 Typový filtr — POŽADAVEK (dnes chybí)
„**Jakou hru** napsal Karel Čapek?" — konceptový termín („hru") je typový filtr
díry, ne druhý `subj`. `known=[(obj,"hru"),(subj,"Karel Čapek")]`, `hole=attr`;
answerer má `_typed_match` (join napsat(X,?) ∧ druh/být(?,hra)).

## 5. Ne-cíle / mantinely
- **Query-side NIKDY nevolá UDPipe** (to je celý smysl). Build grafu UDPipe smí.
- Determinismus (žádné `set`-pořadí v pořadí výstupu).
- Funkční slova a tabulky jen z `jellyai/lang/cs.json` (univerzálnost, jiný
  jazyk = jiný JSON). Žádné hardcode jmen/titulů.

## 5b. Volnost návrhu (DŮLEŽITÉ)
- **Zpětná kompatibilita NENÍ požadavek.** Kód smíš libovolně refaktorovat —
  přejmenovat, přepsat, zahodit `query.py` a postavit znovu, změnit i
  `Pattern`/`SubQuery`/`_pattern_answer`/`_match`, pokud tím vznikne čistší a
  funkčnější řešení. Jediné, co se drží, je **obecná funkčnost** (odpovědi
  z grafu) a **měřený výsledek** (etalon), ne konkrétní API či tvar dat.
- Klidně navrhni pseudo-QL jako plnohodnotný **graphQL-like** jazyk (vlastní
  gramatika/AST), pokud to sedne na graf lépe než `Pattern`. Rozhoduje kvalita
  delivery, ne zachování stávajícího kódu.

## 5c. Autonomní režim, řízený benchmarkem
- **Pracuj samostatně** — neptej se na další krok; iteruj sám.
- **Řiď se benchmarkem.** Každou změnu měř objektivně, ne dojmem:
  - `benchmark/run_etalon.py` — správnost (jádro + gap + `reject`),
  - `benchmark/run_coverage.py` — výtěžnost / zodpověditelnost uzlů,
  - měřicí skript pro `use_templates=True` (kolik jádra šablony samy zvládnou).
- **Rozšiřuj etalon** o nové otázkové tvary při vývoji — každý dořešený gap =
  normativní případ (vč. negativních `reject` a ano/ne), ať graphQL nezregreduje.
- Objektivně měř i **návrh**: srovnávej varianty parseru číslem (kolik jádra,
  kolik false-Pattern místo None). Delivery = číslo, ne tvrzení.
- Po každém kroku: `pytest -q`, oba benchmarky, a když běží web, restart
  (`./jelly web`, port 8080) — běžící instance drží starý kód i graf.

## 6. Akceptační kritéria (definition of done)
1. `.venv/bin/python benchmark/run_etalon.py` s pseudo-QL jako **jedinou**
   query-cestou (bez UDPipe fallbacku na otázky) → **JÁDRO ≥ 24/24** (dnes
   šablony samy 17/24). Cíl je náhrada, ne doplněk — až projde, UDPipe z query
   strany odstranit.
2. `.venv/bin/python -m pytest -q` zelené; `tests/test_query.py` rozšířit o
   ano/ne, typový filtr, dělení běhů, diakritiku, drill data (každý gap = test).
3. `benchmark/run_coverage.py` nezhorší zodpověditelnost uzlů.
4. Interim guardrail: dokud pseudo-QL nedosáhne parity, drž UDPipe jako fallback
   (`use_templates` gate), ať `main` nikdy neklesne pod 24/24.
5. Ruční: „Kdo je jezis?", „Kde se narodil Jezis?", „Kdo byl bratr autora,
   který napsal R.U.R.?" — bez diakritiky i s ní.

## 7. Aktuální gapy (proč 17/24 → co dořešit)
Doloženo měřením (šablony primární):
- víceslovné běhy slepené (Karel Čapek Válku) → **4.3 greedy dělení**,
- „Jakou hru" dá dva subj místo typového filtru → **4.6**,
- ano/ne nerozpoznáno → **4.5**,
- vztahová symetrie/role (bratr) občas vrátí téma místo protistrany → ověřit,
  že relační `Pattern` dá answerera `_match` protistranu (možná stačí
  `hole_role=None` u vztahů),
- reverse „Co se stalo <datum>" a date-drill „v kterém roce" → doplnit
  `date_part` a nechat reverse na answereru.
