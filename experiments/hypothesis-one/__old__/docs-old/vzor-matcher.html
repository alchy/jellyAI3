VZOR matcher — šablony × aktivace

Vyvrcholení celého oblouku: persistentní šablony FAKT-VZOR × QUERY-VZOR × ODPOVĚĎ, nad nimiž běží VZOR matcher — strukturu (stín) najde VZOR, mezi překrývajícími se stíny rozhodne AKTIVACE (světlo z kontextu rozhovoru).

Princip: stín × světlo

Výchozí věta systému: „každá otázka se musí do něčeho trefit, protože je stínem něčeho, co ji generovalo." Dodatek principu stínu:

  • VZOR = STÍN (struktura). Query-VZOR otázky trefí shadow-pár šablony. Vrátí VŠECHNY strukturně shodné → mohou to být překrývající se stíny.
  • AKTIVACE = SVĚTLO (kontext). Slova rozhovoru rozsvítí soubory a entity; mezi překrývajícími se stíny rozhodne, který je právě osvětlený.
odpověď = argmax( vzor_fit(šablona) × aktivace(šablona) )
Neztratit Boha: aktivace drží TÉMA přes stíny — stejný mechanismus, který v koreferenci držel Slovo přes prolog (kotva identity), tady rozhoduje mezi kandidáty odpovědi. Viz Koreference.

Persistentní jednotka — QueryTemplate

Jeden shadow-pár = jeden řádek. Šablony jsou SHARDOVANÉ per zdrojový soubor (data/templates/<doc>.jsonl) — stejně jako korpus — aby se daly NAHRÁT DLE OTÁZKY (jen horké soubory, koncept #60). Append-only (event-log, mergeable, přírůstkově), deduplikace dle id:

{
  "id":         "hash(fact_ref, role, query_vzor)",   // stabilní, dedup
  "fact_vzor":  "PROPN:Nom·VERB:Past·PROPN:Acc·.",     // ŠABLONA FAKTU (díra)
  "query_vzor": "^·PRON:Nom·VERB:Past·?",              // ŠABLONA OTÁZKY
  "role":       "who",                                  // role díry
  "answer":     "Čapek",                                // ODPOVĚĎ (hodnota slotu)
  "surface_q":  "Kdo napsal R.U.R.?",                   // povrch otázky (Ollama)
  "fact_ref":   ["wiki_r.u.r.", 3],                     // provenience (doc, věta)
  "origin":     "ollama"                                // ollama | deterministic
}

To je přesně FAKT-VZOR × QUERY-VZOR × ODPOVĚĎ — samostatné, matchovatelné, s proveniencí.

TemplateStore — VZOR matcher s dynamickým nahráváním

class TemplateStore:
    append(doc, tpl)                      # persistence: šardy per soubor (append-only, dedup id)
    mount(doc_ids)                        # DLE OTÁZKY nahraj jen šablony HORKÝCH souborů (#60)
    unmount(doc_ids)                      # studený soubor = šardy zase zahodit
    match(query_vzor)  → [tpl]            # match jen nad NAMOUNTOVANÝMI (exakt + fuzzy překryv)
    match_fact(fact_vzor) → [tpl]         # druhý směr: strukturální stín
    answer(question, field) → (fakt, ODPOVĚĎ, jistota)
  indexy (build on mount, jen horké):
    by_query_vzor: {vzor → [tpl]}         # klíč matcheru
    by_fact_vzor:  {vzor → [tpl]}

Studený soubor = jen šarda na disku; horký se namountuje. Matcher nikdy nedrží celý korpus šablon — jen to, co otázka rozsvítí. Škáluje na tisíce souborů (past #60 F4).

ActivationField — SPREADING nad váženým grafem vazeb

Ne jen per-uzel teplo — aktivace TEČE po hranách s VÁHOU (spreading / light-beam). To plně využívá graf back-end (mesh, doc_links, shadow-páry, koreference):

class ActivationField:
    nodes: {id → teplo}          # slova / soubory / entity / fakty
    edges: {(a,b) → váha}        # HRANY s vahou (viz „graf vazeb" níže)
    feed(question_lemmas, dataloader)   # rozsvítí slova (tf·idf) + soubory (brána ②)
    spread()                            # teplo teče po hranách × VÁHA (mesh λ^d, doc_links, koref)
    decay()                             # útlum mezi tahy
    weight(tpl) → float                 # aktivace šablony = teplo(doc)+teplo(answer) PO spreadu
    reinforce(tpl)                      # odpověď zpětně přihřeje po hranách (řetěz)

Graf vazeb — hrany a váhy

Vše je uzly + vážené hrany; aktivace se šíří po nich. Většina hran už žije:

Aktivace × VZOR: teplo se rozsvítí ze slov otázky, rozteče po vážených hranách (hot slovo přihřeje sousedy meshem, hot soubor linkované soubory, hot entita příbuzné), a mezi VZOR-kandidáty rozhodne to, které místo grafu je nejvíc osvětlené. Odpověď zpětně přihřeje po hranách → kontext drží řetěz.

Matcher s aktivací a dynamickým nahráním (runtime ⑤)

vstup — živá otázka 
"Co napsal Čapek?"
→ lemmata + query-VZOR
akce — NAHRAJ × VZOR × aktivace 
hot = select_files(lemmata)  # ② brána DLE OTÁZKY
store.mount(hot)             # NAHRAJ jen horké
field.feed(lemmata, …)       # rozsvícení
cands = store.match(q_vzor)  # stín nad namount.
best  = max(cands,
  vzor_fit × field.weight)   # × aktivace

fakt: „…napsal R.U.R."
odpověď: R.U.R.
+ field.reinforce(best)

Krok „NAHRAJ" je klíč: matcher dle otázky nahraje dataset — distinktivní slova otázky rozsvítí soubory (tf·idf brána ②) a mountnou se jen jejich šablony. Detail formátu a indexů: Dynamické nahrávání a formát.

Sjednocení dvou světů

Tohle je to „sloučit dva světy" z počátku:

  • svět rámců/VZORů (přesná struktura, stíny, grafikon) — synth_* + TemplateStore;
  • svět grafu/aktivace (kontext, těžiště, brána souborů, light-beam) — dataloader + ActivationField.

VZOR najde kandidáty, aktivace vybere ten osvětlený. Struktura je přesná (mnoho VZORů, ne rozostření), kontext je měkký (decay, těžiště) — dohromady stín × světlo.
