Fáze ① · Preprocessing

Z surového textu anotovaný korpus

Jediný vstup je čistý text, výstup je anotovaný korpus rozdělený po souborech, na kterém stojí všechny další vrstvy. Anotujeme jednou předem nástrojem UDPipe 2 a uložíme; za běhu dotazu se korpus už jen čte.

Soubor: annotate_corpus.py · třída AnnotateCorpus · konfigurace: blok preprocessing v config.json · výstup: data/corpus/<doc>.pkl (jeden shard na zdrojový soubor).

Příklad: vstup → akce → výstup

Vezmeme jednu větu z dokumentu wiki_r.u.r. a projdeme ji celou fází.

vstup — raw text

Kus souboru data/raw/wiki_r.u.r..txt:

Autor v příběhu varuje před
případnými negativními vlivy
techniky na lidstvo.
akce — UDPipe 2

Plná pipeline (tokenizace → segmentace vět → morfologie → závislostní rozbor) přes REST, výsledek se rozparsuje na tokeny a odvodí se entity.

AnnotateCorpus() \
  .annotate_document(
     "wiki_r.u.r.", raw)
výstup — shard data/corpus/wiki_r.u.r..pkl 
("wiki_r.u.r.", 1): {
 "entities": [],
 "sentences": [[
  {form:"Autor", lemma:"autor",
   upos:"NOUN", deprel:"nsubj",
   head:4, feats:{Case:"Nom"…},
   start:137, end:142}, …]]}

Klíč záznamu je dvojice (doc_id, index_věty). Hodnota má dva klíče: sentences (seznam vět, každá věta je seznam tokenů) a entities (pojmenované entity věty). Token nese tvar, lemma (základní tvar), slovní druh upos, morfologické rysy feats, závislostní hranu deprel → head a znakové offsety start/end.

Názvosloví — surová anotace vs naše klíče: shard nese původní názvosloví UDPipe (deprel = nsubj/obl…, upos = PROPN…, feats = Case/Tense). Vrstva ① je věrná anotace a nenormalizuje — surový deprel je potřeba dál. Převod na naše klíče (who / where / …) dělá až vrstva ③ přes role_key(deprel + pád + předložka) (tabulka deprel_to_role v lang/cs.json) — viz Termíny → Rolový katalog. Proto shard drží surové názvosloví: z deprel + pád se náš klíč odvodí kdykoli, opačně ne.
Díry (placeholdery): zájmena a elidovaný podmět („on", „jeho", Ø) jsou DÍRY — ukazují na entitu nepojmenovanou v klauzuli. Detekují se ÚFAL rysem (ne seznamem tvarů: ho/mu/jemu… ÚFAL lemmatizuje na on) a plní se antecedentem, aby fakt nesl skutečnou entitu — viz Díry — čím je zaplníme.

Proč to tak děláme

  • Anotace jednou, offline. Parsovat každou pasáž za běhu by bylo pomalé a hlavně by to vneslo neuronové rozhodování do runtime. UDPipe 2 je neuronový, ale používá se jen na anotaci předem — v rozhodovací smyčce žádný neuron není.
  • UDPipe 2 místo UDPipe 1. Nová verze snižuje chyby značkování/rozboru o 40–50 %: opravuje např. Kafarnaum (dřív mylně podmět) na nmod a správně rozpozná účelové věty (advcl).
  • Entity z rysu NameType, ne z NameTagu. NameTag lepil jména se sousedy („Ježíš Duchem"). Nově bereme jen souvislé tokeny téhož jména a shodného pádu, takže pádová neshoda span neslepí.
Past, kterou jsme zavřeli: data se do REST posílají přes soubor (-F data=@file). Inline -F data=… tiše ořezával velký víceřádkový vstup na ~485 znaků — bible tak ztrácela 95 % textu (55 vět místo 1151). Vždy přes soubor.

Proč po souborech — fragment (koncept #60)

Anotace se neukládá do jednoho velkého picklu, ale do shardu na každý zdrojový soubor (data/corpus/<doc>.pkl). Každý soubor je samostatná mountovatelná jednotka: za běhu se do grafu nahrají jen ty soubory, které odpovídají rozhovoru (frekvenční analýza tf·idf — četnost × unikátnost termínů), ne celý korpus naráz. Monolitický pickle by tenhle dynamický mount znemožnil.

Souvislost: jak se soubory nahrávají podle matche a jaký index musí každý soubor mít, popisuje Dynamické nahrávání a formát ukládání; co přesně tvoří naši anotační cache popisuje Anotační cache. Samotný tf·idf index a mount staví vrstva ② (dataloader).

Třída AnnotateCorpus

Anotátor korpusu: surové texty ze složky raw_dir → jeden shard na soubor do corpus_dir. Instance drží konfiguraci vrstvy (URL a pinnutý model UDPipe 2, cesty, velikost dávky); čisté transformace jsou statické metody bez stavu, síťové a I/O kroky jsou instanční.

AnnotateCorpus(config_path="config.json")

Konstruktor načte blok preprocessing z JSON konfigurace — nic není natvrdo v kódu (jiný jazyk/nasazení = jiný JSON). Uloží si ho do self.cfg pro všechny další metody.

a = AnnotateCorpus()          # vezme ./config.json
vstupcesta ke config.json (volitelná) výstupinstance s konfigurací vrstvy

run() → None

Projde všechny *.txt ve raw_dir, zanotuje je a uloží jeden shard na soubor do corpus_dir plus _meta.json s pinnutým i skutečně použitým modelem (reprodukovatelnost). Vstupní data nikdy nepřepisuje.

AnnotateCorpus().run()
# [i] 2207260632 : hotovo → …/data/corpus/<doc>.pkl (vět 11926, dokumentů 13)
vstupsložka raw_dir (z configu) výstupdata/corpus/<doc>.pkl (13 shardů) + _meta.json

annotate_document(doc_id, text) → (dict, str|None)

Zanotuje jeden dokument: rozdělí text na dávky po odstavcích, každou pošle do UDPipe 2 a věty čísluje průběžně napříč dávkami. Selhaná dávka se zaloguje jako error a přeskočí, aby zbytek doběhl.

recs, model = AnnotateCorpus().annotate_document(
    "wiki_r.u.r.", open("data/raw/wiki_r.u.r..txt").read())

# výstup: dict o 59 záznamech — jeden záznam:
recs[("wiki_r.u.r.", 1)]
# → {"entities": [],
#    "sentences": [[{"form":"Autor", "lemma":"autor", "upos":"NOUN",
#                    "deprel":"nsubj", "head":4, "start":137, "end":142}, …]]}
model   # → "czech-pdtc-ud-2.17-251125"
vstupdoc_id (str), text (str) výstup{(doc_id, i): {entities, sentences}} + jméno modelu

parse_conllu(text) → (list[list[token]], str|None)  static

Rozparsuje CoNLL-U výstup UDPipe 2 na věty (seznamy tokenů); dělí prázdným řádkem a přeskakuje víceslovné/prázdné uzly. Každému tokenu přidá znakové offsety z TokenRange.

sents, model = AnnotateCorpus.parse_conllu(conllu_str)
sents[0][0]["lemma"], sents[0][0]["start"]   # → ("být", 0)
vstupCoNLL-U (str) výstupvěty jako seznamy tokenů + jméno modelu

entities_from_nametype(tokens) → list[dict]  static

Z rysu NameType jedné věty odvodí entity: Giv/Sur → "P" (osoba), Geo → "G" (místo). Osobní span drží jen souvislé tokeny shodného pádu, takže se jméno neslepí se sousedem v jiném pádu.

AnnotateCorpus.entities_from_nametype(veta_tokeny)
# → [{"text":"Karla Čapka","type":"P","start":112,"end":123}]
vstuptokeny jedné věty (list[dict]) výstupentity {text, type, start, end}

_udpipe2(text, tries=4) → str|None

Pošle text do UDPipe 2 (plná pipeline) přes dočasný soubor a vrátí CoNLL-U, nebo None po vyčerpání pokusů. Soubor kvůli tomu, že inline vstup se ořezává; retry s prodlužující se pauzou kvůli občasné nedostupnosti služby.

conllu = AnnotateCorpus()._udpipe2("Vešel do Egypta.")
# výstup: CoNLL-U (str) — id, tvar, lemma, upos … head, deprel, offsety:
# 1  Vešel   vejít   VERB   …  0  root   TokenRange=0:5
# 2  do      do      ADP    …  4  case   TokenRange=6:8
# 3  Egypta  Egypt   PROPN  …  1  obl    TokenRange=9:15
# 4  .       .       PUNCT  …  1  punct  TokenRange=15:16
vstupraw text jedné dávky (str) výstupCoNLL-U (str) nebo None

_chunks(text, limit) → list[str]  static

Rozdělí text na dávky do limit znaků na hranicích odstavců (prázdný řádek), aby se věta nikdy nerozťala. Odstavec delší než limit jde v jedné dávce sám.

AnnotateCorpus._chunks(text, 40000)   # → ["odstavec\n\nodstavec…", …]
vstuptext (str), limit (int) výstupseznam textových dávek

_token_range(misc) → (int|None, int|None)  static

Ze sloupce MISC v CoNLL-U vytáhne znakové offsety TokenRange=start:end. Když sloupec offset nenese, vrátí (None, None).

AnnotateCorpus._token_range("TokenRange=137:142|SpaceAfter=No")  # → (137, 142)
vstupMISC sloupec (str) výstupdvojice offsetů
