Perzistentní data

Naše anotační cache

Rozbor textu nástrojem UDPipe 2 je pomalý a hlavně neuronový. Uděláme ho proto jednou předem, offline, a výsledek uložíme do vlastní statické cache. Za běhu dotazu se už jen čte — v rozhodovací smyčce žádný neuron není (teze systému drží).

Co cache je a k čemu

Cache je anotovaný korpus rozdělený po souborech. Každý zdrojový soubor má svůj shard; ten je zároveň mountovatelná jednotka pro dynamické nahrávání.

cestaobsahformát
data/corpus/<doc>.pklanotace jednoho souborupickle {(doc,idx): {entities, sentences}}

Formát shardu (jeden záznam)

("wiki_r.u.r.", 1): {
  "entities": [ {"text":"Karla Čapka", "type":"P", "start":112, "end":123} ],
  "sentences": [[
    {"form":"Autor", "lemma":"autor", "upos":"NOUN", "deprel":"nsubj", "head":4,
     "feats":{"Case":"Nom", …}, "start":137, "end":142}, …
  ]]
}
  • klíč (doc_id, index_věty) — provenience až na úroveň věty;
  • sentences — seznam vět, věta = seznam tokenů (tvar, lemma, slovní druh, morfologie, závislostní hrana deprel→head, znakové offsety);
  • entities — pojmenované entity (osoba „P" / místo „G"), odvozené z rysu NameType; podrobně viz Fáze ① Preprocessing.

Jak se přidává nový obsah

Přidání dokumentu je trojkrokové. Vstupní data nikdy nepřepisujeme — přidáváme.

1 — nový raw soubor 
data/raw/muj_text.txt
(doc_id = "muj_text")
2 — anotace ① 
python annotate_corpus.py
# → data/corpus/muj_text.pkl
# + přepíše _meta.json
3 — přepočet statů ② 
Dataloader().build_indexes()
# idf + všechny indexy
# aktuální

Proč krok 3 přepočítává vše: idf (unikátnost termínu) závisí na počtu souborů, takže nový soubor posune váhy i ostatním. build_indexes() je idempotentní nad aktuální množinou shardů — je to jediná „aktualizuj všechny staty" funkce. Krok 2 zanotuje soubory ze data/raw/ a nový soubor si vezme automaticky.

Poznámka: anotace i indexy jsou deterministické — model UDPipe 2 je v configu pinnutý, skóre se vždy řadí. Stejný vstup → stejná cache.
