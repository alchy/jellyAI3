Stav projektu

Deterministický český Q&A systém: ze surového textu postaví mapu faktů a nad ní grafem (tekoucí světlo) + lingvistickými vzory (VZOR) dohledá fakt. Runtime je deterministický (jediná živá služba = UDPipe 2 na rozbor otázky). Kde je jasno, odpoví; kde ne, doptá se; když neví, nehádá — nově i díky naučené assurance bráně.

Vstupní bod pokračování: tato stránka = aktuální stav (metriky, co je hotové, co ne, jak měřit). Směr dál: naučené směrování nad VZORY. Jádro konceptu: dialogový automat.

2. Co je postavené

Pipeline vrstev ①–⑤ (viz rozcestník: preprocessing → dataloader → grammar → synthesis → queryparser). Nad tím tato session přidala:

  • Řetězení kontextu v dialogu (§4b) — navazující otázka zdědí téma, otázka s vlastní entitou přepne. „Kdy se narodil?" po „Kdo je Karel Čapek?" → 1890.
  • Aliasy / identita (stránka) — rozpadlé jméno (Barbora ≡ Němcová) sloučeno → Babička clarify→HIT.
  • is_factual fix — stray „?" v závorce nezahodí definiční větu (Božena → spisovatelka).
  • Kvantifikátor guard — „množství/řada" není faktická odpověď.
  • Polární smyčka VZORŮ (ano/ne) — existence fakt-tvaru → ano/ne. „Je pes šelma? → ano".
  • Kopula hlavní klauzule má přednost před slovesem vedlejší věty (extrakce).
  • Stejnovětná relation-extrakce — vztah (bratr/manžel/otec) jako hrana: „byl bratrem Josefa Čapka" → bratr(Karel, Josef); symetrické obousměrně. relation 0→4/18. Grafový substrát.
  • Naučená assurance brána (směr) — malá NN nad rysy+aktivací: p(správně)<práh → nehádej. Validovaný směr, ale dnes VYPNUTÁ (přeučuje na 76 vzorcích — čeká na víc dat).

3. Architektonický směr

Klíčový obrat session: ruční if-then pravidla nejsou správný systém (množí se → regrese ke kartám; relation-guard to dokázal — regredoval). Směr = malá NN nad maticí VZORŮ + aktivací, učení offline → inference deterministická. Měření ukázalo: routing je minor páka (oracle strop +4/−6), skutečná páka je assurance — a je učitelná. První realizace = assurance brána (už běží). Detail a čísla: naučené směrování nad VZORY.

4. Poctivě — co nefunguje

  • relation / attribute / count = 0 %. Systém tvar otázky nedovede na fakt („Kdo byl bratr X?", „Kolik zubů?"). Chybí VZOR-cesta k těmto fakt-tvarům.
  • Subtilní confident-wrongy brána nechytá — „bratr Karla → spisovatel" (profese z DOMOVSKÉHO souboru vypadá jako platná identita). Chytá jen křiklavé cross-domain (bible pro Čapka).
  • R.U.R. doména — „R.U.R." se tokenizuje na R/U/R → mountne bible; datumy/co-je selhávají. Čeká na node-based retrieval.
  • temporal 36 % — půlka dat se neextrahuje (běžný text mimo bio-závorku).

5. Jak spustit a měřit

ptát se (REPL):        python3 ask.py            # řetězí kontext
živá vizualizace:      .venv/bin/python viz.py   # viewBase :8080 (+ panel dokumentů)
velký etalon (145):    python3 eval_large.py     # mode-aware, po kategoriích
malý etalon (25):      python3 eval_answers.py
trénink brány:         python3 train_gate.py     # → gate.json (gate_data.json se dogeneruje)
přestavba dat:         python3 annotate_corpus.py → dataloader.build_indexes()
                       → synth_registry.py → build_facts_all.py → build_templates_all.py

Data (korpus/index/fakty/šablony) jsou gitignored (reprodukovatelné z raw + build_*). Na deploy je nutné je přestavět. gate.json je commitnutý artefakt (malé váhy).

6. Další kroky

  1. Chytit subtilní confident-wrongy — jemnější typové/doménové rysy brány (typ entity, cross-doc vzdálenost); potřebují čistší data.
  2. Hygiena fakt-store → odemkne auto-gen tréninku (dnes moc zašuměný, po filtru 31 čistých).
  3. Odemknout nulové kategorie — relation/count VZOR-cestou (query-tvar → fakt-tvar).
  4. R.U.R. / doména — node-based retrieval (entita = uzel s fakty).
